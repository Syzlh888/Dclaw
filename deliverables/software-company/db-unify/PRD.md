## 1. 产品目标与定位

### 1.1 产品定位

DB-Unify 是一款面向数据库运维人员和管理者的**多数据库统一 SQL 执行与结果聚合平台**。核心解决"同一 SQL 需在多个数据库重复执行、结果分散难以对比"的痛点。

### 1.2 产品目标

| # | 目标 | 衡量标准 |
|---|------|----------|
| G1 | **批量执行效率** | 同一 SQL 在 50 个数据库执行，操作步骤从 50 次降到 1 次 |
| G2 | **结果可读性** | 执行结果支持聚合视图与横向对比，关键差异秒级定位 |
| G3 | **层级化组织** | 数据库按 平台→类型→区县→医院 四层架构组织，勾选操作符合心智模型 |

## 2. 用户画像与使用场景

### 2.1 用户画像

| 属性 | 描述 |
|------|------|
| 角色 | 数据库运维工程师 / 信息平台管理员 |
| 技能 | 熟悉 SQL，日常使用 Navicat/DBeaver 等工具 |
| 痛点 | 管理数十个数据库，同一 SQL 反复切换连接执行；结果散落各窗口，无法快速对比 |
| 期望 | 一处编写 SQL → 勾选目标库 → 一键执行 → 聚合查看结果 |

### 2.2 核心使用场景

| # | 场景 | 描述 |
|---|------|------|
| S1 | 批量数据巡检 | 每天早晨在所有医院库执行健康检查 SQL，汇总异常 |
| S2 | Schema 一致性校验 | 发布前检查各库表结构是否一致，找出差异字段 |
| S3 | 数据统计汇总 | 统计各区县医院的业务数据量，生成汇总报表 |
| S4 | 应急数据修复 | 发现数据问题后，在选定范围的库中批量执行修复 SQL |
| S5 | 按层级筛选执行 | 只对"电子病历前置库"或"兰山区"的医院执行特定 SQL |

## 3. 核心功能需求

## 4. 层级架构数据库树详细设计

### 4.1 四层树结构

```
📁 临沂市全民健康信息平台                          ← 第1层：平台
├── 📁 电子病历前置库                              ← 第2层：前置库类型
│   ├── 📁 兰山区                                  ← 第3层：区县
│   │   ├── 🏥 兰山区人民医院                       ← 第4层：医院（实际数据库）
│   │   └── 🏥 兰山区妇幼保健院
│   └── 📁 沂水县
│       ├── 🏥 沂水县人民医院
│       ├── 🏥 沂水县中医医院
│       └── 🏥 沂水县妇幼保健院
└── 📁 健康档案前置库
    ├── 📁 兰山区
    │   ├── 🏥 兰山区人民医院
    │   └── 🏥 兰山区妇幼保健院
    └── 📁 沂水县
        ├── 🏥 沂水县人民医院
        ├── 🏥 沂水县中医医院
        └── 🏥 沂水县妇幼保健院
```

**数据模型**：

```
Platform (1层)
  └── PreDbType (2层)
       └── District (3层)
            └── Hospital (4层) → 关联实际数据库连接信息
```

| 层级 | 实体 | 字段 | 说明 |
|------|------|------|------|
| L1 | Platform | id, name | 平台名称 |
| L2 | PreDbType | id, platform_id, name | 前置库分类 |
| L3 | District | id, predb_type_id, name | 区县 |
| L4 | Hospital | id, district_id, name, db_connection_id | 医院，关联数据库连接 |

### 4.2 勾选逻辑

#### 交互规则

| 操作 | 行为 |
|------|------|
| 勾选 L4 节点 | 仅选中该医院数据库 |
| 勾选 L3 节点 | 自动选中该区县下所有医院，父节点显示 ✓ |
| 勾选 L2 节点 | 自动选中该前置库类型下所有医院，父节点显示 ✓ |
| 勾选 L1 节点 | 选中平台下所有医院（本例 = 全选） |
| 部分子节点被勾选 | 父节点显示半选状态（indeterminate） |
| 取消勾选父节点 | 所有子节点取消勾选 |

#### 勾选状态传播

```
子节点变更 → 向上冒泡计算父节点状态（全选✓ / 半选- / 未选☐）
父节点变更 → 向下广播设置所有子节点状态
```

#### 勾选计数

- 树顶部显示：**已选中 X / 共 Y 个数据库**
- 每个 L2/L3 节点旁显示：`(2/5)` 格式的已选/总数

### 4.3 搜索过滤

| 功能 | 说明 |
|------|------|
| 搜索框 | 位于数据库树顶部，支持按医院名称、区县名称搜索 |
| 实时过滤 | 输入即过滤，未匹配节点隐藏，匹配节点自动展开其父路径 |
| 高亮 | 匹配文本高亮显示 |
| 清空 | 一键清空搜索恢复完整树 |

## 6. 批量执行引擎需求

### 6.1 执行流程

```
用户编写 SQL → 勾选目标数据库 → 点击执行
     ↓
引擎依次/并行连接各数据库
     ↓
每个连接独立执行 SQL
     ↓
收集结果 → 实时更新状态面板 → 全部完成后切换到结果视图
```

### 6.2 执行策略

| 策略 | 说明 | 默认值 |
|------|------|--------|
| 并发数 | 同时执行的最大数据库连接数 | 5 |
| 单库超时 | 单个数据库执行超时时间 | 30s |
| 失败策略 | 单库失败后是否继续执行其他库 | 继续执行 |
| 重试次数 | 连接失败后自动重试次数 | 1 |

### 6.3 执行状态面板

| 状态 | 图标 | 说明 |
|------|------|------|
| 等待中 | ⏳ | 排队等待执行 |
| 执行中 | 🔄 | 正在执行 SQL |
| 成功 | ✅ | 执行成功，可查看结果 |
| 失败 | ❌ | 执行失败，显示错误信息 |
| 超时 | ⏱️ | 执行超时，已自动终止 |

### 6.4 状态面板布局

```
┌─ 执行状态 ──────────────────────────────────────────┐
```

## 7. 结果查看与对比需求

### 7.1 聚合视图（默认视图）

将所有库的查询结果合并到一张表中，每行标注数据来源。

```
┌─ 聚合视图 ────────────────────────────────────────────────────────┐
│ 筛选: [全部库 ▾]  搜索: [________]                                │
├──────────┬──────────────┬────────────────┬───────────────┤
│ 来源库   │ hospital_name │ patient_count  │ date          │
├──────────┼──────────────┼────────────────┼───────────────┤
│ 兰山区人民医院(病历) │ 兰山区人民医院  │ 128            │ 2025-07-14   │
│ 兰山区妇幼保健院(病历)│ 兰山区妇幼保健院 │ 56            │ 2025-07-14   │
│ 沂水县人民医院(病历)  │ 沂水县人民医院   │ 203            │ 2025-07-14   │
│ ...     │ ...          │ ...            │ ...           │
└──────────┴──────────────┴────────────────┴───────────────┘
```

**聚合规则**：
- 各库结果必须有相同的列结构（列名+类型一致），不一致时提示
- 来源列自动添加，格式为 `医院名(前置库类型)`
- 支持按来源列筛选和排序

### 7.2 对比视图

选取 2-5 个库，横向并排展示结果，差异高亮。

```
┌─ 对比视图 ───────────────────────────────────────────────────────┐
│ 对比库: [兰山区人民医院 ▾] [沂水县人民医院 ▾] [沂水县中医医院 ▾]  │
├────────────┬──────────────┬──────────────┬──────────────┤
│ 字段        │ 兰山区人民医院 │ 沂水县人民医院 │ 沂水县中医医院 │
├────────────┼──────────────┼──────────────┼──────────────┤
│ table_name  │ 🟡 users     │ users        │ users        │
│ column_count│ 🔴 15        │ 18           │ 🔴 15        │
│ index_count │ 🟡 3         │ 4            │ 4            │
└────────────┴──────────────┴──────────────┴──────────────┘
```

**对比规则**：
- 相同值：正常显示
- 不同值：橙色高亮（🟡）
- 仅部分库存在：红色高亮（🔴）
- 支持按字段切换行/列视角

### 7.3 单库详情视图

点击单个库可查看完整结果表格，等同于传统 SQL 客户端的查询结果展示。

### 7.4 导出功能

| 导出方式 | 说明 | 优先级 |
|----------|------|--------|
| CSV 导出 | 单库或全量导出为 CSV | P1 |
| Excel 导出 | 多 Sheet 导出，每个库一个 Sheet | P1 |
| 复制行 | 选中行直接复制到剪贴板 | P1 |
| SQL 导出 | 导出执行语句+参数，便于复现 | P2 |

## 8. 非功能需求

### 8.1 性能

| 指标 | 要求 |
|------|------|
| 数据库树加载 | ≤ 1s（50 个节点） |
| 并发执行 | 支持 10 个数据库同时执行 |
| 结果渲染 | 单库 ≤ 10,000 行结果，渲染 ≤ 2s |
| 聚合结果 | 50 库各 200 行结果，聚合渲染 ≤ 5s |
| SQL 编辑器 | 输入延迟 ≤ 50ms |

### 8.2 安全

| 要求 | 说明 |
|------|------|
| 连接信息加密 | 数据库密码使用 AES-256 加密存储，不明文落盘 |
| SQL 类型限制 | 只读模式下严格过滤 DML/DDL |
| 操作日志 | 记录每次执行的 SQL、目标库、执行人、时间（P2 存储） |
| 会话管理 | 连接超时自动断开，不维持长连接池 |

### 8.3 可靠性

| 要求 | 说明 |
|------|------|
| 执行容错 | 单库失败不影响其他库执行 |
| 连接重试 | 连接失败自动重试 1 次 |
| 结果缓存 | 执行结果保留在内存，会话内可反复查看 |

### 8.4 可用性

| 要求 | 说明 |
|------|------|
| 初始学习成本 | 有 SQL 基础的用户 5 分钟内完成首次查询 |
| 错误提示 | 执行失败时显示具体错误原因和建议 |
| 响应式布局 | 支持数据库树/编辑器/结果面板比例调整 |

## 10. 待确认问题

| # | 问题 | 影响范围 | 建议 |
|---|------|----------|------|
| Q1 | 是否需要支持多种数据库类型（MySQL/PostgreSQL/Oracle 等）？ | 执行引擎、连接管理 | MVP 先支持 MySQL，后续扩展 |
| Q2 | 数据库连接信息由谁录入？是否有批量导入方式？ | 数据库树管理 | 支持手动添加 + CSV 批量导入 |
| Q3 | 执行 DDL/DML 时是否需要二次确认？ | 安全性 | 强制弹出确认框，显示影响范围 |
| Q4 | 是否需要支持 SQL 参数化（如日期参数替换）？ | 编辑器 | P1 支持，用 `{{param}}` 语法 |
| Q5 | 层级结构是否需要支持自定义层级数（非固定4层）？ | 数据模型 | MVP 固定4层，P2 考虑自定义 |
| Q6 | 结果数据是否需要持久化存储？ | 架构 | MVP 仅内存保存，P2 支持持久化 |
| Q7 | 是否需要多用户/权限管理？ | 安全 | MVP 单用户，P1 多用户角色 |
| Q8 | 50 个数据库是否全部为同类型（MySQL）？ | 连接管理 | 待用户确认，影响驱动选择 |
